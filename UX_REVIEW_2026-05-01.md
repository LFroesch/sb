# sb UX Review

Date: 2026-05-01

Context: review notes after reading the current TUI/Foreman/Agents surface with the intended product bar in mind: this should feel lower-friction than using raw Codex/Claude sessions directly, not like a more complicated wrapper around them.

## Core Read

The strongest parts of the app are real:

- Foreman is a meaningful differentiator.
- The job list plus review loop is a useful control plane.
- The app already does real work instead of being a thin launcher.

The current UX problem is not that the app has "too many features" in the abstract. The problem is that the main path often feels like operating the implementation instead of operating the product.

The Agents surface is where this shows up most.

## Main Friction Points

### 1. Too much hidden mode/focus state

The app relies heavily on internal mode switches and focus switches:

- page
- mode
- launchFocus
- attachedFocus
- agentManageFocus
- edit/select submodes inside launch/manage flows

That makes the interaction powerful, but it also means the user has to keep asking:

- "what screen am I actually on?"
- "what does enter do right now?"
- "is this typing mode or navigation mode?"

That is exactly the kind of friction this product is supposed to remove.

### 2. The launch flow still feels more like a control matrix than a composer

The current New Run flow is better than an earlier raw setup surface, but it still has the feel of:

- tab through many concerns
- sometimes press `e` to enter a typed selector
- sometimes use arrows/j/k
- sometimes `enter` means toggle
- sometimes `enter` means advance
- sometimes `enter` means launch

This is efficient once learned, but not calm. For a product positioned as "less session juggling," this should feel more like a short guided compose flow and less like navigating a dashboard of overrides.

### 3. Agent Setup is still structurally awkward

This is already called out in `WORK.md`, and that callout is correct.

The issue is not just wording. The surface is conceptually mixed:

- library management
- object editing
- wizard/group progression
- advanced composition
- selector/editor submodes

It works, but it does not feel settled. It reads like a persistence editor for internal data models, not a crisp user-facing setup tool.

### 4. Keybind load is too high on the high-value surface

The help surface is broad enough that it starts to signal a problem by itself. On Agents alone the user has to learn:

- list actions
- filter actions
- launch actions
- attach actions
- session-control actions
- Foreman actions
- setup-management actions

Power users can handle this, but the product promise here is not "a very capable TUI if you memorize it." The promise is lower operational friction.

### 5. Attached sessions still feel a bit split-brain

The app supports both tmux-backed jobs and exec-chat jobs, which is useful. But the user experience still carries that implementation split more than it should.

The user mental model should be:

- open a run
- read what happened
- type if the run is live
- send a control action if needed
- back out cleanly

Right now that model exists, but the interaction details still feel slightly different depending on runner type and focus state.

### 6. Some doc/product signals are drifting

There is at least one stale README reference to docs that are not present in this checkout. That matters because this project depends heavily on product language being coherent. When the docs drift, the UI wording is harder to trust too.

## What Seems Correct Already

These are directions that seem worth keeping:

- Foreman as part of the core product, not a side feature
- task-backed launches inheriting repo automatically
- explicit distinction between normal runs and queued-for-Foreman runs
- attached session controls using literal actions instead of fake abstractions
- keeping advanced setup in-app instead of forcing raw JSON for everything

I would not throw those away.

## Where The Earlier Plan Was Too Aggressive

The previous implementation direction I suggested pushed too hard toward a broad structural rewrite of the Agents flow.

That is risky for two reasons:

1. The current system already contains a lot of real product learning.
2. A large UX rewrite could easily burn time without proving which specific frictions matter most in practice.

A better next step is probably to tighten the current shape rather than replace it wholesale.

## More Conservative Direction

### 1. Make the primary path more obvious without rewriting everything

Keep the existing architecture if needed, but make the default user path much more explicit:

- list jobs
- start a run
- choose task source or freeform
- confirm setup
- attach/review later

If a user can stay on that path without learning the advanced controls, the app will feel much better immediately.

### 2. Reduce hidden branching inside New Run

Without redesigning the whole screen, the main goal should be:

- fewer cases where `enter` changes meaning unexpectedly
- less reliance on `e` opening special selector state
- stronger on-screen explanation of the current step and next action

This is probably the highest-leverage UX improvement in the codebase.

### 3. Demote Agent Setup from "part of the run path" to "advanced config area"

The app should still have Agent Setup, but it should feel clearly secondary. A user launching a run should not feel like they are one wrong keypress away from falling into schema-edit territory.

That means:

- stronger separation in wording
- lower visibility in the primary action row
- less setup jargon leaking into launch flow language

### 4. Unify attached-session behavior more aggressively

Even if the backends differ, the visible interaction contract should be as close as possible between tmux and exec-chat sessions.

The product should feel like "resume this run," not "pick which runner-specific UI rules apply now."

### 5. Prune visible keybinds before pruning capabilities

I would be careful about deleting functionality early.

A safer move is:

- keep capabilities
- expose fewer actions at once
- make the footer ruthlessly contextual
- let the help screen carry the long tail

That preserves power while reducing the feeling of overload.

## Suggested Next Passes

If I were continuing this tomorrow, I would do the work in this order:

1. Audit the New Run flow end-to-end and remove the most confusing focus/state transitions.
2. Tighten footer/help copy on Agents so only the current high-value actions are visible.
3. Simplify Agent Setup presentation without changing the data model underneath.
4. Normalize attached-session behavior and wording across tmux vs exec-chat.
5. Fix stale doc references and make the product language more consistent.

## Short Version

The app is not failing because the idea is wrong. It is failing in spots because too much of the implementation model is visible to the operator.

The main UX target should be:

"make the common path feel obvious and calm, while leaving the advanced machinery available when needed."

That is narrower than a full rewrite, and probably a better fit for the current state of the code.
